# User Data Table Plan

Status: draft for review before Supabase migrations are created.

This plan covers data owned by, generated by, or private to an app user. Shared
learning content (`words`, `questions`, `levels`, and related tables) remains
covered by `DATA_DESIGN.md`.

## 1. Current project findings

The app currently treats `User` as one aggregate containing profile, duck
power, level progress, assessment radar, streak, props, and onboarding state.
That is useful at the Kotlin/UI layer, but these values should not be stored in
one database row because they have different update patterns and retention
requirements.

The existing SQL draft in `docs/CONTENT_BRIEF.md` already proposes:

- `profiles`
- `level_progress`
- `streaks` and `checkins`
- `user_props`
- `mistake_words`
- `practice_sessions` and `practice_answers`
- `assessment_results`
- `onboarding_profiles`
- `scratch_card_log`
- `feedback` and `question_error_reports`

The product specification additionally requires persisted settings, consent
history, avatar unlocks, linked account state, and account-deletion requests.

## 2. Recommended ownership model

Use Supabase Auth as the identity authority:

- `auth.users` owns authentication identity, verified phone/email, and provider
  identities.
- `public.profiles.id` equals `auth.users.id` and owns app-visible profile data.
- Every private child table has a direct `user_id`, even when the user could be
  discovered through another foreign key. This makes RLS simple and auditable.
- The Kotlin `User` object is assembled by the repository from several tables;
  it is not a one-table database model.

Do not store raw verification codes, provider access tokens, passwords, or
other authentication secrets in `public`.

## 3. MVP tables

### 3.1 Identity and profile

#### `profiles`

One row per authenticated user.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid PK` | FK to `auth.users(id)` with cascade delete |
| `public_user_code` | `text unique` | Stable user-facing ID; do not display the raw auth UUID |
| `nickname` | `text` | Required; enforce product nickname rules |
| `avatar_path` | `text null` | Selected system avatar or uploaded avatar path |
| `duck_power` | `int` | Non-negative lifetime total |
| `onboarding_status` | enum/text | `not_started`, `in_progress`, `completed`, `skipped` |
| `created_at` | `timestamptz` | Default `now()` |
| `updated_at` | `timestamptz` | Maintained by trigger |

Do not duplicate `phone` in this table unless the app needs a separately
masked display value. The verified phone belongs to Supabase Auth.

#### `user_settings`

One row per user for app-controlled preferences.

| Column | Type | Default |
|---|---|---|
| `user_id` | `uuid PK` | FK to `profiles` |
| `learning_reminder_enabled` | `bool` | `false` |
| `streak_reminder_enabled` | `bool` | `false` |
| `reminder_timezone` | `text` | User/device IANA timezone |
| `sound_enabled` | `bool` | `true` |
| `haptics_enabled` | `bool` | `true` |
| `updated_at` | `timestamptz` | `now()` |

OS notification and microphone permission status should be read from the
device, not treated as authoritative database state.

#### `user_consents`

Append-only evidence of agreement to legal documents.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid PK` | |
| `user_id` | `uuid` | FK to `profiles` |
| `document_type` | `text` | e.g. `terms`, `privacy`, `carrier_terms` |
| `document_version` | `text` | Exact accepted version |
| `accepted_at` | `timestamptz` | |
| `withdrawn_at` | `timestamptz null` | Where withdrawal is applicable |

Unique active acceptance should be enforced for
`(user_id, document_type, document_version)`.

### 3.2 Learning position and assessment

#### `level_progress`

One row per user and level.

Recommended additions to the current draft:

- `best_star_rating smallint`
- `completed_session_count int`
- `unlocked_at timestamptz`
- `completed_at timestamptz`
- checks for level progress `0..1`, stars `0..3`, and non-negative counts

Unique key: `(user_id, level_number)`.

The visible current level is the highest unlocked/in-progress level. It should
be derived, not duplicated in `profiles`.

#### `assessment_results`

One immutable row per completed assessment.

Prefer explicit numeric columns for the five radar axes:

- `overall_ielts_score`
- `vocabulary_score`
- `listening_score`
- `speaking_score`
- `reading_score`
- `writing_score`
- `recommended_level_number`
- `report jsonb`
- `assessment_version`
- `taken_at`

All ability values require checks in the `0..10` range. “Previous” radar values
should be derived from the preceding assessment rather than copied into every
row.

#### `onboarding_profiles`

One row per user:

- `user_id`
- `questionnaire_version`
- `answers jsonb`
- `started_at`
- `completed_at`
- `skipped_at`

JSONB is appropriate while the questionnaire is small and versioned. Store
stable answer codes, not translated display strings.

### 3.3 Practice history

#### `practice_sessions`

One row per started practice run:

- `user_id`
- `level_number`
- `card_id` or stable activity key
- `session_type` (`daily`, `mistake_review`, `assessment`, `challenge`)
- `started_at`, `completed_at`
- `correct_count`, `total_count`
- `star_rating`
- `base_power`, `combo_bonus`, `speed_bonus`, `duck_power_earned`
- `status` (`started`, `completed`, `abandoned`)

Keeping reward components makes scoring auditable when rules change.

#### `practice_answers`

One row per answered question:

- `user_id`
- `session_id`
- `question_id`
- `answer_given`
- `is_correct`
- `time_ms`
- `combo_after_answer`
- `answered_at`

Add `user_id` directly. The current draft's RLS policy references this column,
but the column does not exist. Also enforce that an answer's `user_id` matches
its session owner, preferably through a controlled RPC for session completion.

Persist the session, its answers, score changes, level progress, mistakes, and
check-in in one database transaction/RPC. Separate client writes can leave
partial state after a network failure.

### 3.4 Streaks and rewards

#### `streaks`

One current-state row per user:

- `current_days`
- `longest_days`
- `goal_days`
- `last_checkin_date`
- `protection_used_on date null`
- `updated_at`

#### `checkins`

Append-only daily history with unique `(user_id, checkin_date)`.

Use a constrained `source`, such as `practice`, `review`, or `challenge`.
Streak state should be updated server-side from this log to prevent duplicate
increments when a user completes multiple sessions in one day.

#### `user_props`

Current inventory with unique `(user_id, prop_type)`, `count >= 0`, and
per-prop maximum checks where applicable.

#### `reward_transactions`

Replace the narrow `scratch_card_log` with a general append-only reward ledger:

- `user_id`
- `source_type` and `source_id`
- `reward_type`
- `amount`
- `metadata jsonb`
- `created_at`

Scratch cards, practice bonuses, challenge rewards, avatar unlocks, and future
promotions can then use the same idempotent audit trail.

### 3.5 Mistake review

#### `mistake_words`

Use typed columns instead of placing the scheduling fields inside
`review_state jsonb`:

- `user_id`, `word_id`
- `review_stage smallint` with check `0..5`
- `wrong_count int`
- `correct_review_count int`
- `last_wrong_at`
- `last_reviewed_at`
- `next_due_at`
- `mastered_at null`
- `created_at`, `updated_at`

Keep mastered rows for learning history instead of deleting them immediately.
The active notebook query filters `mastered_at is null`.

## 4. Feature-support tables

These can follow the core MVP migration but are required by the written product
specification.

### `user_avatar_unlocks`

Unique `(user_id, avatar_key)`, with `unlocked_at` and `source_transaction_id`.
System avatar definitions should live in a shared reference table or app
configuration.

### `push_devices`

One row per app installation/device:

- `user_id`
- `installation_id`
- `platform`
- `push_token`
- `timezone`
- `app_version`
- `last_seen_at`
- `is_active`

Tokens must be rotated and invalid tokens disabled. This table is private and
should not be returned with the normal profile aggregate.

### `account_deletion_requests`

Supports the specified delayed deletion workflow:

- `user_id`
- `reason_code`
- `reason_text`
- `requested_at`
- `scheduled_for`
- `status` (`pending`, `cancelled`, `processing`, `completed`, `rejected`)
- `processed_at`

Submitting a request should disable normal login/use through server-side
account state. Final deletion must remove the Auth user so FK cascades remove
private app data.

### `feedback`

Keep the existing table, but constrain:

- category to the documented set
- body length to 1,000 characters
- workflow status/severity to known values

User insert and read-own access is appropriate. Status changes should be
service-role/admin only.

### `question_error_reports`

Add `details`, `status`, and an optional `session_id`. Users may insert and read
their own reports; only staff may moderate them.

## 5. RLS and write rules

1. Enable RLS on every public user table.
2. User reads use `user_id = auth.uid()`; `profiles` uses `id = auth.uid()`.
3. Do not grant unrestricted client `UPDATE` on reward balances, duck power,
   streak counters, assessment scores, or completion state.
4. Use security-definer RPCs with a fixed `search_path` for:
   - completing a practice session
   - granting/consuming rewards
   - recording a check-in
   - submitting account deletion
5. Admin moderation fields on feedback/reports are service-role only.
6. Storage policies must restrict uploaded avatars to a user-owned path such as
   `avatars/{auth.uid()}/...`.

## 6. Required indexes

In addition to primary and unique keys:

- `level_progress (user_id, is_unlocked, level_number)`
- `practice_sessions (user_id, started_at desc)`
- `practice_answers (user_id, answered_at desc)`
- `practice_answers (session_id)`
- `mistake_words (user_id, mastered_at, next_due_at)`
- `checkins (user_id, checkin_date desc)`
- `assessment_results (user_id, taken_at desc)`
- `reward_transactions (user_id, created_at desc)`
- `feedback (status, created_at desc)`

## 7. Migration order

1. Enums, reusable timestamp trigger, and validation functions.
2. `profiles` plus automatic profile creation after Auth signup.
3. `user_settings`, `user_consents`, and `onboarding_profiles`.
4. `level_progress` and `assessment_results`.
5. `practice_sessions`, `practice_answers`, and the transactional completion
   RPC.
6. `mistake_words`, `streaks`, and `checkins`.
7. `user_props`, `reward_transactions`, and avatar unlocks.
8. Push devices, feedback/reporting, and account deletion.
9. RLS tests using two users plus anonymous/service-role cases.
10. Kotlin database DTOs and Supabase repository implementations.

## 8. Decisions to confirm before SQL

- Whether phone-only login is the first release or whether WeChat/QQ/Apple
  account linking must ship immediately.
- Whether users may upload custom avatars or only select unlocked system
  avatars.
- Whether practice answer history is deleted, anonymized, or archived after
  90 days.
- Whether a pending deletion request can be cancelled during the seven-day
  window.
- Whether reminders are fixed at 12:00/20:00 or configurable per user.
